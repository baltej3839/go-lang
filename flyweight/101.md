Flyweight is one of the most misunderstood patterns because its purpose is very specific:

> **Share common data between many objects to save memory.**

The key question is:

> **"Am I creating thousands or millions of similar objects that contain repeated information?"**

---

# First Principles

Suppose you're making a game.

You have:

```text
1,000,000 trees
```

Each tree contains:

```text
Type = Oak
Texture = oak.png
Color = Green
Height = 10m
Position = x,y
```

Without Flyweight:

```text
Tree1
    Texture = oak.png
Tree2
    Texture = oak.png
Tree3
    Texture = oak.png
...
```

You're storing:

```text
oak.png
```

a million times!

---

# Split the data

Separate:

## Intrinsic State (shared)

```text
Type = Oak
Texture = oak.png
Color = Green
```

Same for every oak tree.

---

## Extrinsic State (unique)

```text
Position = (10,20)
Position = (30,40)
Position = (50,80)
```

Different for each tree.

---

# Memory Diagram

Without Flyweight

```text
Tree1
  texture oak.png

Tree2
  texture oak.png

Tree3
  texture oak.png
```

Three copies.

---

With Flyweight

```text
                OakTreeType
             texture oak.png
             color green
                  ▲
                  │
        ┌─────────┼─────────┐
        │         │         │
      Tree1     Tree2     Tree3
    pos(1,2)  pos(3,4)  pos(5,6)
```

Only one copy of:

```text
texture oak.png
```

# In Go

Suppose you have:

## Shared Object

```text
CharacterStyle
```

containing:

```text
font
size
color
```

---

## Individual Character

containing:

```text
position
actual character
pointer to style
```

Thousands of characters can share one style.

# Flyweight Factory

Usually there is a cache:

```text
map[string]*Flyweight
```

When someone asks:

```text
"Oak"
```

Factory checks:

```text
Already exists?
```

If yes:

```text
Return existing object.
```

Otherwise:

```text
Create and store it.
```

Suppose a million characters use:

```text

12
Black
```

Without Flyweight:

```text
'A'

'B'
font=Arial

'C'

```

Huge waste.

Instead:

```text
Style
 font=Arial
 size=12
 color=Black
```

and every character references it.

# When to Use Flyweight

When:

### Many objects exist

```text
100,000+
```

and

### Most fields are identical

```text
texture
font
metadata
configuration
```

and

### Memory matters

---

# Structure

Usually:

```text
Flyweight
```

contains shared state.

```text
FlyweightFactory
```

stores:

```text
map[key]*Flyweight
```

Clients store:

```text
unique data
+
pointer to flyweight
```

---

# Example Mental Model

Suppose 10,000 students use:

```text
Punjab University
```

Instead of every student storing:

```text
University Name
Address
Logo
```

you store:

```text
University object
```

once.

Each student stores:

```text
Roll Number
Name
Pointer to University
```

---

# Difference from Prototype

Prototype:

```text
Copy objects.
```

Flyweight:

```text
Share objects.
```

---

# Difference from Singleton

Singleton:

```text
One object globally.
```

Flyweight:

```text
Many shared objects in a cache.
```

Example:

```text
Oak
Pine
Birch
```

Three flyweights, not one.

---

# Difference from Factory

Factory:

```text
Creates objects.
```

Flyweight Factory:

```text
Creates OR reuses objects.
```

# Rule of Thumb

Whenever you think:

> "I'm storing the same data thousands of times."

Think:

```text
Flyweight
```

and ask:

```text
What part is unique?
What part can be shared?
```

That separation between **extrinsic state** (unique) and **intrinsic state** (shared) is the heart of the Flyweight pattern.
