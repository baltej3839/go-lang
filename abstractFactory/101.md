Abstract Factory is easiest to understand if you first think:

> A Factory creates **one object**.
>
> An Abstract Factory creates a **family of related objects**.

---

# Problem

Suppose your application needs:

1. A database
2. A logger

For development:

```text
MemoryDB
ConsoleLogger
```

For production:

```text
PostgresDB
ZapLogger
```

Without Abstract Factory:

```go
db := &PostgresDB{}
logger := &ConsoleLogger{}
```

Oops!

You accidentally mixed:

```text
Production DB
+
Development Logger
```

---

# Idea

Group related objects together.

Think:

```text
Development Environment
    ↓
MemoryDB
ConsoleLogger

Production Environment
    ↓
PostgresDB
ZapLogger
```

---

# Step 1: Define Interfaces

Database:

```go
type Database interface {
    Save()
}
```

Logger:

```go
type Logger interface {
    Log(msg string)
}
```

---

# Step 2: Concrete Implementations

Development:

```go
type MemoryDB struct{}

func (m *MemoryDB) Save() {}

type ConsoleLogger struct{}

func (c *ConsoleLogger) Log(msg string) {}
```

Production:

```go
type PostgresDB struct{}

func (p *PostgresDB) Save() {}

type ZapLogger struct{}

func (z *ZapLogger) Log(msg string) {}
```

---

# Step 3: Abstract Factory Interface

```go
type EnvironmentFactory interface {
    CreateDatabase() Database
    CreateLogger() Logger
}
```

Read it like:

```text
An environment knows how to create
all dependencies belonging to it.
```

---

# Step 4: Development Factory

```go
type DevFactory struct{}
```

```go
func (d *DevFactory) CreateDatabase() Database {
    return &MemoryDB{}
}

func (d *DevFactory) CreateLogger() Logger {
    return &ConsoleLogger{}
}
```

---

# Step 5: Production Factory

```go
type ProdFactory struct{}
```

```go
func (p *ProdFactory) CreateDatabase() Database {
    return &PostgresDB{}
}

func (p *ProdFactory) CreateLogger() Logger {
    return &ZapLogger{}
}
```

---

# Usage

```go
var factory EnvironmentFactory

factory = &ProdFactory{}

db := factory.CreateDatabase()
logger := factory.CreateLogger()
```

Result:

```text
db      -> PostgresDB
logger  -> ZapLogger
```

Switch:

```go
factory = &DevFactory{}
```

Result:

```text
db      -> MemoryDB
logger  -> ConsoleLogger
```

No other code changes.

---

# Visual Diagram

Normal Factory:

```text
NewDatabase()
      ↓
PostgresDB
```

Creates one object.

---

Abstract Factory:

```text
ProdFactory
      │
      ├── PostgresDB
      └── ZapLogger

DevFactory
      │
      ├── MemoryDB
      └── ConsoleLogger
```

Creates families of objects.

# Real Example

Suppose your booking app needs:

```text
Store
Cache
Logger
```

Development:

```text
MemoryStore
MemoryCache
ConsoleLogger
```

Production:

```text
PostgresStore
RedisCache
ZapLogger
```

Abstract Factory:

```go
type InfraFactory interface {
    CreateStore() Store
    CreateCache() Cache
    CreateLogger() Logger
}
```

Then:

```go
factory := &ProdFactory{}

store := factory.CreateStore()
cache := factory.CreateCache()
logger := factory.CreateLogger()
```

Everything belongs to the same environment.

---

# Is Abstract Factory Common in Go?

Not really.

Go developers often prefer:

```go
func NewProductionDependencies() Dependencies
```

where:

```go
type Dependencies struct {
    Store Store
    Cache Cache
    Logger Logger
}
```

Example:

```go
deps := NewProductionDependencies()

service := NewBookingService(
    deps.Store,
    deps.Cache,
    deps.Logger,
)
```

This is basically Abstract Factory without explicitly saying so.

---

# When to use Abstract Factory

Ask yourself:

```text
Am I creating multiple related objects together?
```

If:

```text
Database
Cache
Logger
```

must all come from:

```text
Development
OR
Production
```

then Abstract Factory is useful.

---

# Difference Summary

### Factory

Creates one object:

```go
db := NewDatabase()
```

---

### Builder

Configures one object:

```go
server := NewServerBuilder().
    TLS(true).
    Timeout(30).
    Build()
```

---

### Prototype

Copies one object:

```go
copy := original.Clone()
```

---

### Abstract Factory

Creates a family of objects:

```go
factory := &ProdFactory{}

db := factory.CreateDatabase()
logger := factory.CreateLogger()
cache := factory.CreateCache()
```

---

### My advice

Don't memorize the pattern. Remember the question:

```text
Do several objects belong together
and need to change together?
```

If yes, think:

```text
Abstract Factory
```

This pattern becomes much more interesting once you learn Dependency Injection, because they are often used together in large Go applications.
