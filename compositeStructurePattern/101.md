Composite is one of the most beautiful patterns because it lets you treat:

```text
One thing
```

and

```text
A group of things
```

the same way.

---

# First Principles

Suppose you're building a file system.

You have:

```text
File
Folder
```

A folder contains:

```text
Files
Folders
```

Example:

```text
Documents
│
├── Resume.pdf
├── Notes.txt
└── Projects
      ├── main.go
      └── README.md
```

You'd like to do:

```go
documents.Show()
```

and have everything displayed recursively.

---

# Problem without Composite

You'd need:

```go
ShowFile()

ShowFolder()
```

and folders would need special logic for files vs folders.

---

# Composite Idea

Make both implement the same interface.

```go
type Node interface {
    Show()
}
```

---

# Leaf

A file is a leaf.

```go
type File struct {
    Name string
}

func (f *File) Show() {
    fmt.Println(f.Name)
}
```

---

# Composite

A folder contains nodes.

```go
type Folder struct {
    Name     string
    Children []Node
}
```

and:

```go
func (f *Folder) Show() {
    fmt.Println(f.Name)

    for _, child := range f.Children {
        child.Show()
    }
}
```

Notice:

```go
Children []Node
```

not:

```go
[]File
[]Folder
```

because children can be either.

---

# Usage

```go
resume := &File{Name: "Resume.pdf"}
notes := &File{Name: "Notes.txt"}

projects := &Folder{
    Name: "Projects",
    Children: []Node{
        &File{Name: "main.go"},
        &File{Name: "README.md"},
    },
}

documents := &Folder{
    Name: "Documents",
    Children: []Node{
        resume,
        notes,
        projects,
    },
}
```

Then:

```go
documents.Show()
```

prints:

```text
Documents
Resume.pdf
Notes.txt
Projects
main.go
README.md
```

---

# Why is this called Composite?

Because:

```text
Folder
```

is composed of:

```text
Files
Folders
```

and they all satisfy:

```go
Node
```

```text

             []Node

```

Notice the recursion:

```text
Folder contains Nodes
```

# Composite in Go

The heart of the pattern is:

```go
type Component interface {
    Operation()
}
```

Leaf:

```go
type Leaf struct{}
```

Composite:

```go
type Composite struct {
    children []Component
}
```

Recursive call:

```go
for _, child := range children {
    child.Operation()
}
```

# Real-Life Exercise

Build an organization hierarchy.

Interface:

```go
type Employee interface {
    Work()
}
```

Leaf:

```text
Developer
Designer
Tester
```

Composite:

```text
Manager
```

Manager contains:

```go
team []Employee
```

Calling:

```go
ceo.Work()
```

should recursively call:

```text
CEO managing...

CTO managing...

Developer coding...
Developer coding...

Design lead managing...

Designer designing...
Tester testing...
```

# When to think Composite

Whenever you hear:

```text
Tree
Hierarchy
Nested objects
Parent-child relationship
Recursive structure
```

think:

```text
Composite
```

Examples:

* Filesystem
* HTML DOM
* Menus
* AST (compiler)
* Organization chart
* Categories and subcategories
* Comments with replies
* Folders and files

---

## Rule of Thumb

If you want:

```go
thing.Do()
```

and whether `thing` is:

* one object

or

* a group of objects

shouldn't matter,

then you're probably looking at the **Composite pattern**.
