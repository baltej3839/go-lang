Adapter is one of the most useful patterns in Go because you're constantly integrating with libraries, APIs, databases, and third-party services.

# First Principle

Suppose your code expects:

```text
A
```

but you have:

```text
B
```

and you can't change either.

You need:

```text
A ← Adapter ← B
```

The Adapter translates between them.

---

# Real-Life Example

Suppose your application understands:

```go
Notifier
```

with:

```text
Send(message)
```

But a third-party library provides:

```text
SendSMS(phone, text)
```

You can't change the third-party library.

You don't want your code to know about:

```text
phone numbers
special parameters
weird API names
```

So you create:

```text
SMSAdapter
```

that implements your interface and internally calls the third-party API.

---

# Structure

## Target Interface

What your application expects:

```text
Notifier
```

---

## Adaptee

What already exists:

```text
TwilioClient
```

with some incompatible methods.

---

## Adapter

Something that:

```text
Implements Notifier
```

and internally uses:

```text
TwilioClient
```

# Why Go Loves Adapters

Interfaces are small.

Suppose:

```go
type Logger interface {
    Log(string)
}
```

If some library already has:

```go
Info(string)
```

you simply wrap it.

Go programmers often don't even call it "Adapter"; they just call it:

```text
wrapper
```

# When to Use Adapter

Whenever you hear:

### "I already have something."

and

### "Its interface doesn't match what I need."

```text
Adapter
```

---

# Difference from Factory

Factory:

```text
Creates objects.
```

Adapter:

```text
Makes incompatible objects work together.
```

---

# Difference from Composite

Composite:

```text
Treat group and individual uniformly.
```

Adapter:

```text
Translate one interface into another.
```

---

# Difference from Strategy

Strategy:

```text
Different algorithms behind one interface.
```

Adapter:

```text
One implementation translated into another interface.
```

---

# Real Production Example

Suppose your service expects:

```text
EmailSender
```

But AWS SES SDK gives:

```text
SendEmailWithContext()
```

You don't spread AWS-specific code everywhere.

Instead:

```text
SESAdapter
```

implements:

```text
EmailSender
```

and internally talks to AWS.

This way, switching to SendGrid later means only changing the adapter.

---

# Exercise (No Code)

Imagine your application expects:

```text
Storage
    Save(data)
    Load(id)
```

But you need to use:

### Redis Library

with methods:

```text
Set(key, value)
Get(key)
```

and later

### S3 Library

with methods:

```text
PutObject()
GetObject()
```

Create:

```text
RedisAdapter
S3Adapter
```

that both satisfy:

```text
Storage
```

and allow your application to remain completely unaware of Redis or S3.

---

### Rule of Thumb

If you say:

> "I wish this thing had the methods my code expects."

then you're looking at an **Adapter**.
