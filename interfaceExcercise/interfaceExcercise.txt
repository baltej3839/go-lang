Perfect. You're past the beginner stage of interfaces now.

At this point, you should stop doing `Animal`, `Dog`, `Cat` exercises and start doing exercises that mirror real Go services.

Here are some progressively harder production-style exercises.

---

# Exercise 1: User Repository

You're building a user service.

The service needs to:

```text
Create User
Get User
Delete User
```

Today data is stored in:

```text
PostgreSQL
```

Later:

```text
MongoDB
```

Later:

```text
In-Memory Storage (for tests)
```

### Task

Design:

```text
UserService
UserRepository
PostgresRepository
MongoRepository
MemoryRepository
```

Questions:

* Which type stores the interface?
* What methods belong in the interface?
* Why shouldn't UserService know about Postgres?

---

# Exercise 2: Cache Layer

# Exercise 11 (Most Important)

Build a complete Booking System.

Entities:

```text
BookingService
BookingRepository
PaymentProcessor
Notifier
```

Possible implementations:

```text
PostgresBookingRepo
MemoryBookingRepo

StripeProcessor
RazorpayProcessor

EmailNotifier
SMSNotifier
```

Flow:

```text
Create Booking
      |
      +--> Save Booking
      |
      +--> Charge Payment
      |
      +--> Send Notification
```

### Goal

Make BookingService depend only on interfaces.

This exercise combines almost everything you've learned:

```text
Service Struct
      |
      +--> Repository Interface
      |
      +--> Payment Interface
      |
      +--> Notification Interface
```

---

If you're targeting production Go backend roles, I would do them in this order:

```text
1. User Repository
2. File Upload System
3. Message Queue
4. Payment Retry Strategy
5. Booking System
```

Those five patterns show up constantly in real Go codebases.
