Bridge is one of the harder patterns to understand because at first it looks very similar to Strategy. The key idea is:

> **Separate two dimensions that can vary independently.**

---

# First Principles

Suppose you're drawing shapes.

You have:

```text
Circle
Square
Triangle
```

and you also have:

```text
Red
Blue
Green
```

Without Bridge, you might end up with:

```text
RedCircle
BlueCircle
GreenCircle

RedSquare
BlueSquare
GreenSquare

RedTriangle
BlueTriangle
GreenTriangle
```

Number of classes:

```text
3 shapes × 3 colors = 9 types
```

As you add more:

```text
10 shapes × 5 colors = 50 types
```

Explosion!

---

# Bridge Idea

Separate:

```text
Shape
```

from

```text
Color
```

and connect them through composition.

---

# Visual

Without bridge:

```text
RedCircle
BlueCircle
GreenCircle
```

With bridge:

```text
Circle ----> Color
Square ----> Color
Triangle --> Color

Color:
    Red
    Blue
    Green
```

Both hierarchies vary independently.

---

# Example

Shape hierarchy:

```text
Circle
Square
```

Color hierarchy:

```text
Red
Blue
```

Then:

```text
Circle + Red
Circle + Blue

Square + Red
Square + Blue
```

are formed dynamically instead of creating separate types.

---

# Why is it called Bridge?

Because:

```text
Shape
```

is connected to

```text
Color
```

through a bridge:

```text
Shape ---------> Color
```

---

# Real Example: Notification

Suppose you have:

### Message Types

```text
Alert
Marketing
Reminder
```

and

### Delivery Methods

```text
Email
SMS
Push
```

Without bridge:

```text
AlertEmail
AlertSMS
AlertPush

MarketingEmail
MarketingSMS
MarketingPush

ReminderEmail
ReminderSMS
ReminderPush
```

9 combinations.

With Bridge:

```text
Alert ------> EmailSender
Marketing --> EmailSender
Reminder ---> EmailSender

Alert ------> SMSSender
Marketing --> SMSSender
...
```

No explosion.

# Real Go Backend Example

Suppose:

### Storage

```text
UserStore
OrderStore
BookingStore
```

and databases:

```text
Postgres
MongoDB
Redis
```

Without bridge:

```text
PostgresUserStore
MongoUserStore
RedisUserStore

PostgresOrderStore
MongoOrderStore
RedisOrderStore
```

Too many structs.

Bridge:

```text
UserStore ------> Database
OrderStore -----> Database
BookingStore ---> Database
```

where Database can be:

```text
Postgres
Mongo
Redis
```

# Structure

Bridge has two hierarchies:

## Abstraction

High-level thing:

```text
Shape
Notification
Store
```

---

## Implementor

Low-level implementation:

```text
Color
Sender
Database
```

The abstraction holds:

```go
implementor Implementor
```

and delegates work to it.

---

# Difference from Strategy

This confuses everyone.

## Strategy

```text
One object
+
One interchangeable algorithm
```

Example:

```text
PaymentService
    |
PaymentStrategy
```

Only strategy varies.

---

## Bridge

Two independent dimensions vary:

```text
Shape varies
Color varies
```

or

```text
Notification type varies
Delivery mechanism varies
```

Bridge prevents combinatorial explosion.

---

# Difference from Adapter

Adapter:

```text
Makes incompatible interfaces work together.
```

Bridge:

```text
Separates two independent hierarchies.
```

---

# Difference from Composite

Composite:

```text
Tree structure.
```

Bridge:

```text
Two orthogonal dimensions.
```

---

# When to Think Bridge

Whenever you see:

```text
A × B
```

and you're creating:

```text
A1B1
A1B2
A2B1
A2B2
...
```

you should think:

```text
Bridge
```

because you probably want:

```text
A ------> B
```

instead.

## Rule of Thumb

If you're seeing:

```text
ThingAThingB
ThingAThingC
ThingDThingB
ThingDThingC
```

and the number of types keeps multiplying,

you're probably looking at a **Bridge pattern**.
